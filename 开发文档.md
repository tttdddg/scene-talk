# 最终交付目标

项目名称：

## SceneTalk——能看见的 AI 对话助手

一句话介绍：

> SceneTalk 通过摄像头理解用户当前看到的场景，听取用户的语音问题，并用文字和自然语音作出连续回答。

最终必须形成以下完整流程：

```text
开启摄像头
    ↓
用户点击语音按钮并提问
    ↓
浏览器识别用户语音
    ↓
自动截取当前摄像头关键帧
    ↓
前端压缩图片
    ↓
后端调用视觉模型
    ↓
返回简洁、准确的回答
    ↓
页面展示回答并语音播报
    ↓
用户继续追问
```

最终演示至少覆盖：

1. 摄像头实时预览；
2. 中文语音提问；
3. 自动截取当前画面；
4. 物品和颜色识别；
5. 场景描述；
6. 基于上下文的连续追问；
7. 文字回答；
8. 中文语音播报；
9. 图片压缩率和请求耗时展示；
10. 权限拒绝、请求失败等异常提示。



# 重要技术边界

## 2.1 DeepSeek-V4 的用途

DeepSeek-V4 只用于驱动 Claude Code：

- 生成项目代码；
- 修改代码；
- 执行测试；
- 检查构建错误；
- 帮助整理 README 和 PR 描述；
- 辅助代码审查。

不要默认使用 DeepSeek-V4 处理摄像头图片。

应用运行时需要另外调用支持图片输入的视觉模型。

## 2.2 视觉模型选择

优先方案：

```text
阿里云百炼 OpenAI 兼容接口
```

模型通过环境变量配置：

```env
VISION_API_KEY=your_api_key
VISION_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
VISION_MODEL=qwen3.5-plus
```

如果控制台中没有该模型，选择当前账号能够调用、明确支持图片输入的视觉模型，并只修改 `VISION_MODEL`。

代码中禁止写死模型名称。

## 2.3 视觉输入方式

不连续上传视频流。

使用：

```text
实时摄像头预览
+
每次用户提问结束后截取一张关键帧
```

这样既能满足“AI 看到摄像头中的内容”，又可以：

- 降低推理费用；
- 减少网络传输；
- 减少模型调用；
- 降低开发复杂度；
- 提高 Demo 稳定性。

设计文档中要主动解释这是端云协同和成本控制设计，不要把它描述成“来不及实现视频流”。
